# Versioning & Release Pipeline

## Version Format

`v<major>.<minor>.<patch>[-suffix]`

Examples: `v4.0.1`, `v4.1.0-alpha`, `v4.2.0-beta`, `v5.0.0-rc1`

The `v` prefix is **required** — the release workflow triggers on the `v*` tag pattern.
Suffix is optional; use `-alpha` / `-beta` / `-rc` for pre-releases.

## Branch → Release Pipeline

```
feat/<name> branch
  → PR opened
      └─ build-linux.yml runs automatically (tests + Linux/Windows builds)
  → PR merged to main
  → git tag vX.Y.Z[-suffix]     ← release trigger
  → git push origin vX.Y.Z[-suffix]
      └─ release.yml runs:
           build-linux  ──┐
           build-windows ─┤ (needs: build-linux)
           create-release ┘ (needs: both) → draft release on GitHub
  → manually review & publish the draft release on GitHub
```

## Tagging a Release

Run these after merging to main:

```bash
git checkout main && git pull
git tag v4.0.1-alpha
git push origin v4.0.1-alpha
```

Then go to **GitHub → Releases** and publish the draft once the workflow completes.

## What CI Produces

**`build-linux.yml`** (PR + push to main):
- Runs frontend tests (Vitest) + Go tests
- Builds Linux binary + Windows binary + NSIS installer (artifacts only, not released)
- PR artifacts auto-expire after 3 days

**`release.yml`** (tag push):
- Builds `KeyLint-vX.Y.Z-linux-amd64` and `KeyLint-vX.Y.Z-windows-amd64.exe`
- Generates `latest.json` for the in-app updater (`UpdaterService.CheckForUpdate`)
- Creates a **draft** GitHub Release with auto-generated notes from PR titles
- All three files are attached to the release

## How AppVersion Is Injected

`main.AppVersion` (in `main.go`) defaults to `"dev"`. Both workflows set it at build time:

```bash
VERSION=$(git describe --tags --always)
go build -ldflags="-X main.AppVersion=${VERSION}" ...
```

`git describe --tags --always` produces:
- `v4.0.1-alpha` — on a clean tag
- `v4.0.1-alpha-3-gabcdef` — 3 commits after the tag (CI only, never release)
- `v4.0.0-5-gabcdef-dirty` — uncommitted changes (build-linux.yml dev builds)

The version is surfaced in Settings → About via `UpdaterService.GetVersion()`.

## What CI Does NOT Do

- No automatic version bump commits
- No `CHANGELOG.md` updates — release notes are auto-generated by GitHub from merged PR titles
- No auto-publish — draft must be manually reviewed and published
- No branch protection enforcement (set that up in GitHub repo settings if needed)
